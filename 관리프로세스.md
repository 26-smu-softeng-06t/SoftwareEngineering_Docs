# 관리 프로세스

본 문서는 TOKKI 팀의 Scrum 운영, Issue/PR 통제, 개발 절차, 문서 관리 기준을 정리한다.

## 1. 계획 프로세스 (Agile/Scrum)

| 항목 | 내용 |
|------|------|
| 스프린트 주기 | 1주 |
| 스탠드업 | 수업 후 5~10분 — 한 일 / 할 일 / 블로커 |
| 추가 미팅 | 필요 시 금요일 비대면 — 스프린트 계획·회고 |
| 스프린트 계획 | 백로그 선정 → 작업 배분 → 완료 기준 합의 |
| 스프린트 회고 | Keep / Problem / Try |

Scrum은 본 프로젝트의 공식 운영 프레임워크이며, 요구사항 단위는 User Story/PBI로 관리한다. 각 PBI는 수용 기준(AC)을 가져야 하며, 완료 여부는 DoD와 PR 검증 결과를 기준으로 판단한다.

## 2. 통제 프로세스

관리 도구는 GitHub Project, GitHub Issue, GitHub Pull Request를 기준으로 한다.

기본 흐름:

1. Issue 등록 시 GitHub Project 백로그에 추가한다.
2. Assignee가 작업을 맡으면 상태를 Ready 또는 In Progress로 변경한다.
3. 작업 중 변경 사항, 블로커, 범위 변경은 Issue 코멘트에 남긴다.
4. 작업 종료 후 PR을 생성하고 템플릿을 채운다.
5. 리뷰/검증 기준을 만족하면 대상 브랜치로 머지한다.

### Dev Branch

- 작업 브랜치 대상은 `dev`이다.
- PR에 대해 Copilot Review를 수행한다.
- lint 통과, 핵심 흐름 수동 확인, 이슈 링크가 있으면 self-merge 가능하다.

### Main Branch

- `main`은 안정 브랜치이며 직접 push를 금지한다.
- `dev -> main` 반영은 별도 PR로 수행한다.
- Triage Review 후 과반(2인+Copilot) 이상 동의 시 머지한다.

## 3. 이슈 관리 프로세스

이슈 로그는 GitHub Issue에 직접 등록한다.

현재 제공되는 Issue Template:

1. Bug Report
2. Documentation
3. Meeting / Decision Record
4. PBI / User story
5. Refactor / Tech debt
6. Spike / Research
7. Feature

주요 라벨:

- `Backend`, `Fullstack`, `Design`, `test`
- `priority/high`, `priority/medium`, `priority/low`
- `needs-triage`, `Blocked`, `question`, `help wanted`
- `pbi`, `sprint`, `task`, `refactor`, `spike`, `Decision`

이슈 제목 권장 형식:

```text
[type] 짧은 문제/요청 설명
```

이슈 본문에는 배경, 목표, 범위, 완료 조건(DoD), 참고자료를 포함한다.

## 4. 개발 프로세스

### 요구사항 도출 및 명세

브레인스토밍 → 기능/비기능 요구사항 분류 → `요구사항명세서.md` 반영 → 팀 전체 검토 → Issue/PBI로 추적한다.

요구사항 변경은 다음 절차를 따른다.

1. Issue 또는 PR에서 변경 필요성을 설명한다.
2. 관련 FR/NFR, User Story, AC 영향을 확인한다.
3. 승인 후 `요구사항명세서.md`, `User_Story.md`, 관련 다이어그램을 갱신한다.
4. 확정 의사결정이면 `decisions.md`에 DEC 항목을 추가한다.

### 방법론 및 분석 도구 적용

- Scrum 이벤트(Planning, Daily Scrum, Review, Retrospective)를 중심으로 운영한다.
- 요구사항 단위는 User Story/PBI로 관리하고 수용 기준으로 완료 여부를 판단한다.
- SRS 기준 문서는 `요구사항명세서.md`로 고정한다.
- 일정 계획은 WBS를 기준으로 작성하고, PERT 추정으로 기대 소요시간과 임계경로를 검증한다.
- MVP 범위 산정은 FP(Function Point) 스타일 기능 분해를 참고한다.

### 설계 산출물

| 산출물 | 도구/위치 |
| --- | --- |
| WBS | 팀 관리 도구 |
| Use Case / Class Diagram | draw.io / PlantUML, `Use_Case_Diagram.md`, `Class_Diagram.md` |
| ERD | draw.io 또는 Mermaid, `ERD.md` |
| Sequence Diagram | Mermaid, `Sequence_Diagram.md` |
| UI 설계서 | Google AI Studio 또는 Figma |

## 5. 개발 환경 및 저장소 구조

TOKKI 저장소는 2026-04-28 기준으로 다음 구조를 사용한다.

- `frontend/`: Vite React 클라이언트
- `backend/`: Spring Boot API 서버
- `scripts/`: 로컬/운영 보조 스크립트
- `.env.example`: 공유 가능한 환경변수 템플릿
- `.env`: 개인 로컬 비밀값 파일, Git 커밋 금지

상세 구조와 환경변수 기준은 `개발환경_및_구조.md`를 따른다.

## 6. AI Agent / Instruction 운영

- 공통 지시문은 `AGENTS.md`를 기준으로 한다.
- Claude Code, Copilot, Cursor, Gemini 등 도구별 지시문은 공통 지시문을 참조하는 얇은 wrapper로 유지한다.
- `Onboarding` 폴더에는 프롬프트, instruction, `.env` 템플릿이 포함될 수 있으므로 공개 문서에 원문을 복제하지 않는다.
- 실제 비밀값, 개인 프롬프트, 토큰, DB 접속 문자열은 문서 저장소와 Wiki에 기록하지 않는다.

## 7. Definition of Done

작업은 다음 조건을 만족할 때 완료로 본다.

- 요구사항 또는 수용 기준 충족
- 관련 코드 리뷰 반영 완료
- 테스트 또는 수동 검증 결과 기록
- 필요한 문서 업데이트 완료
- 관련 Issue/PR 링크 정리
- 비밀값이 커밋되지 않았는지 확인
